# 知识管理与智能问答系统需求分析

## 1. 背景与目标

本需求分析基于两份新增需求说明书：

- [`docs/knowledge_management_system.md`](../knowledge_management_system.md)：《知识管理系统 — 需求说明书》
- [`docs/smart_quote_system.md`](../smart_quote_system.md)：《智能问答系统需求说明书》

并补充参考现有系统级文档：

- [`docs/system.md`](../system.md)：技术监督辅助平台总需求说明。
- [`docs/report_generation_system.md`](../report_generation_system.md)：报告生成系统需求说明书。

并结合已确认的补充决策整理：

- 会话历史使用服务端数据库持久化。
- 知识库页面对所有登录用户可见；普通用户只允许查看 `public` 知识库。
- 向量数据库使用 Qdrant。
- 原文下载需要权限校验和审计日志。
- 未来会提供问答准确率考核标准，包括标准问题集、标准答案和评分规则。
- 需要支持图片 OCR。
- 模型服务需要支持多供应商。
- 报告生成暂按后续模块处理。

系统目标是建设一个面向电力行业的知识管理与智能问答平台，形成从知识沉淀、文档处理、向量检索、智能问答、引用溯源到质量评测的完整闭环。知识管理模块负责知识资产的组织、处理和检索底座；智能问答模块基于知识管理产生的文本切片和向量索引，提供面向用户的自然语言问答能力。

从 `docs/system.md` 看，完整平台还包括报告生成能力。本文档仍以知识管理和智能问答为主，但会补充它们与报告生成、素材管理、模板管理之间的数据依赖和接口边界，避免后续模块接入时重复建设知识检索、模型配置和文档处理能力。

## 2. 系统边界

### 2.1 本期纳入分析范围

- 用户注册、登录、会话保持和角色权限。
- 知识库创建、编辑、删除、分类、可见性和检索策略配置。
- 文档上传、标签、删除、批量操作、状态跟踪、失败重试和切片查看。
- 素材文档上传、键值对标签、组合过滤和后续报告生成数据支撑。
- 文档解析、图片 OCR、扫描件 OCR、语义切片、向量化和 Qdrant 入库。
- 前台知识检索，包括跨知识库检索、标签过滤、阈值过滤和重排序。
- 智能问答，包括意图识别、RAG、多轮对话、SSE 流式输出、非流式输出和服务端历史会话。
- 引用溯源，包括原文片段、来源文档、相关性分数、文档下载、权限校验和审计。
- 管理后台配置，包括 LLM、Embedding、Rerank、OCR、解析器和问答检索参数。
- 数据统计与监控。
- 问答准确率评测能力预留。
- 与报告生成模块的衔接，包括素材、模板、检索、模型配置和历史数据复用。

### 2.2 暂不展开的后续模块

- 报告生成的完整交互实现，包括大纲编辑、逐章节生成、DOCX 导出、报告记录管理和模板可视化编辑。
- 复杂组织架构下的部门级、多租户级知识权限。
- 完整的配置版本回滚 UI。
- 完整的模型效果评测平台 UI。

以上内容可以在数据模型和接口边界中预留扩展点，但不作为当前需求分析的核心流程展开。

### 2.3 文档关系与合并口径

现有文档之间存在层级关系：

- `docs/knowledge_management_system.md` 描述知识库、文档、解析、检索和模型配置，是知识底座的直接来源。
- `docs/smart_quote_system.md` 描述智能问答、RAG、SSE、引用溯源、思考过程和问答管理，是智能问答能力的直接来源。
- `docs/system.md` 是系统总说明，补充了分段策略、素材管理、非流式问答、数据分析意图、报告生成意图、引用交互细节和报告生成整体位置。
- `docs/report_generation_system.md` 是后续报告生成模块的独立说明，本文只提取它对知识管理和智能问答的依赖，不展开成完整报告生成设计。

因此本文档采用的合并口径是：知识管理和智能问答的功能需求完整展开；报告生成只作为下游能力说明它需要哪些知识资产、素材资产、检索能力和模型配置支持。

## 3. 用户角色与权限

系统至少包含三类角色。

| 角色 | 可见范围 | 主要能力 |
| --- | --- | --- |
| 普通用户 | 仅可查看 `public` 知识库 | 登录、查看公开知识库、前台检索、发起问答、查看个人历史会话、查看有权限的引用来源 |
| 管理员 | 可查看所有知识库 | 管理知识库、管理文档、配置检索参数、配置模型、测试检索效果、查看统计 |
| 超级管理员 | 可查看所有知识库 | 拥有管理员能力，并可管理系统级配置、用户角色和高风险操作 |

权限控制要求：

- 所有 API 请求都必须进行身份认证。
- 未登录用户不得访问系统业务功能。
- 管理后台仅允许管理员和超级管理员访问。
- 普通用户可以访问知识库页面，但只能看到 `public` 知识库。
- 检索和问答不能只依赖前端隐藏数据，后端必须按用户权限过滤知识库范围。
- 原文下载必须进行文档级权限校验。
- 高风险操作需要记录审计日志，包括文档下载、删除、配置变更、处理任务重试等。

## 4. 认证与会话

### 4.1 登录认证

系统应支持用户名和密码注册、登录。登录成功后应建立会话状态，页面刷新后无需重复登录。

认证能力包括：

- 用户注册。
- 用户登录。
- 用户登出。
- 当前用户信息获取。
- 会话过期处理。
- 管理员角色识别。

### 4.2 智能问答会话历史

智能问答的会话历史使用服务端数据库持久化，而不是浏览器本地存储。

功能要求：

- 用户可以创建新会话。
- 用户可以切换历史会话。
- 用户可以删除自己的会话。
- 页面刷新后自动恢复最近会话或会话列表。
- 用户换设备登录后仍可访问自己的历史会话。
- 每条消息需要保存角色、内容、时间、关联引用、使用的配置快照等信息。

前端本地仅适合保存临时状态，例如当前选中的会话 ID、未发送草稿、UI 展开状态等。

## 5. 知识库管理

知识库是系统的核心业务对象，不只是文件夹。每个知识库需要承载分类、可见性、文档集合、分段策略、检索策略和权限边界。

### 5.1 知识库基础信息

系统应支持：

- 创建知识库。
- 编辑知识库。
- 删除知识库。
- 批量删除知识库。
- 查看知识库详情。
- 查看知识库列表。

知识库基础字段包括：

- 名称。
- 描述。
- 文档类型。
- 可见性。
- 分段策略。
- 检索策略。
- 文档数量。
- 切片数量。
- 创建人。
- 创建时间。
- 更新时间。

### 5.2 知识库分类

知识库应支持按文档类型分类，包括但不限于：

- 规程规范。
- 技术报告论文。
- 术语条目。
- 通用文档。

分类用于列表筛选、检索范围选择、问答策略配置和统计分析。

### 5.3 知识库可见性

知识库应包含 `visibility` 字段。

建议值：

- `public`：普通用户、管理员、超级管理员均可查看和检索。
- `private`：仅管理员和超级管理员可查看和检索。

后续可扩展：

- `department`：部门范围可见。
- `restricted`：指定用户或角色可见。

当前确认规则：

- 知识库页面对所有登录用户可见。
- 普通用户只允许查看 `public` 知识库。
- 管理员和超级管理员可查看所有知识库。

### 5.4 知识库列表能力

知识库列表需要支持：

- 按类型过滤。
- 按可见性过滤。
- 按关键词搜索。
- 分页。
- 展示文档数量、创建时间、创建人等元信息。
- 批量选择和批量删除。

### 5.5 分段策略配置

知识库应支持配置文档分段策略。分段策略会影响后续检索召回和问答准确率。

建议支持：

- 基于标题层级智能分段。
- 按固定字符数分段。
- 固定长度分段下配置切片长度。
- 固定长度分段下配置重叠长度。
- 自定义分隔符。
- 递归合并模式。

当分段策略变更时，系统应自动触发后台重新处理该知识库下所有就绪状态文档。

### 5.6 检索策略配置

知识库应支持配置检索策略：

- 语义向量检索。
- 向量检索 + 重排序。

检索参数包括：

- Top K。
- 相似度阈值。
- 是否启用重排序。
- 重排序 Top N。
- 重排序分数阈值。

当检索策略变更影响索引或切片处理方式时，系统应触发必要的重新处理或重新索引。

## 6. 文档管理

### 6.1 文档上传

系统应支持向指定知识库上传文档。

上传方式：

- 拖拽上传。
- 点击选择文件上传。

文件类型包括但不限于：

- PDF。
- DOCX。
- PPTX。
- XLSX。
- MD。
- TXT。
- 常见图片格式。

上传时应支持添加标签，用于后续检索过滤和文档管理。

### 6.2 文档标签

文档标签用于元数据过滤和业务分类。

要求：

- 上传时添加标签。
- 上传后编辑标签。
- 支持按标签过滤检索。
- 支持多标签组合过滤。

标签可以先采用普通字符串数组；如果后续需要更强的分类能力，可扩展为键值对标签，例如 `专业=锅炉`、`年份=2025`、`来源=技术监督`。

### 6.3 文档状态

文档处理应具备完整状态流转。

| 状态 | 含义 |
| --- | --- |
| 已上传 | 文件已保存，尚未开始处理 |
| 解析中 | 正在从文件中提取文本或结构化内容 |
| OCR 中 | 正在对图片或扫描件执行 OCR |
| 切片中 | 正在根据分段策略生成文本切片 |
| 向量化中 | 正在生成 embedding 并写入 Qdrant |
| 就绪 | 文档已可用于检索和问答 |
| 失败 | 处理失败，需要查看错误原因或重试 |

原需求中没有单列 `OCR 中`，但由于已确认需要支持图片 OCR，建议在状态模型中显式体现，便于用户理解处理进度。

### 6.4 文档列表与操作

文档列表应支持：

- 分页浏览。
- 按状态筛选。
- 按标签筛选。
- 按关键词搜索。
- 查看处理状态。
- 查看失败原因。
- 单个删除。
- 批量删除。
- 失败后重试处理。
- 就绪文档查看切片详情。

删除文档时，需要同步处理：

- 关系型数据库中的文档记录。
- 文档切片记录。
- Qdrant 中对应向量。
- 原始文件存储。
- 关联引用和审计记录的保留策略。

## 7. 素材管理

`docs/system.md` 将素材管理放在知识管理范围内，报告生成说明也明确管理后台需要素材管理页面。素材管理和普通知识库文档相似，但它更偏向报告生成、模板填充和专业材料复用，应作为知识资产的一类专门管理对象。

### 7.1 素材范围

系统应支持电厂专业素材文档上传和管理。

素材分类至少包括：

- 厂级专业报告。
- 技术文档。
- 检查报告。

素材可以复用文档上传、解析、OCR、切片和向量化处理管线，但在业务上需要保留素材分类、模板关联、报告生成引用等额外元数据。

### 7.2 素材标签

素材应支持标签化管理，并优先考虑键值对标签。

示例：

- `电厂=某电厂`
- `专业=锅炉`
- `年份=2025`
- `报告类型=迎峰度夏检查报告`
- `来源=厂级专业报告`

功能要求：

- 增加标签。
- 删除标签。
- 修改标签。
- 按标签组合条件过滤素材列表。
- 支持标签用于素材检索和报告生成素材匹配。

### 7.3 素材操作

素材管理应支持：

- 上传素材文档。
- 查看素材列表。
- 按分类过滤。
- 按标签组合过滤。
- 编辑标签信息。
- 删除素材。
- 查看素材处理状态。
- 查看素材切片详情。

### 7.4 与知识库文档的关系

素材管理不应重复建设一套完全独立的文件处理系统。

建议关系：

- 文件保存、解析、OCR、切片、embedding、Qdrant 入库复用文档处理管线。
- 素材业务元数据单独维护，例如素材分类、适用报告类型、关联模板、专业、电厂、年份。
- 素材可被普通知识检索命中，但是否进入普通用户可见范围需要受知识库可见性和素材权限控制。
- 报告生成模块可以基于素材标签、模板映射和向量检索召回相关素材。

## 8. 文档解析、OCR 与处理管线

### 8.1 处理主流程

文档处理管线应覆盖：

1. 文件上传并保存原文。
2. 创建文档记录和处理任务。
3. 解析文件为文本或结构化内容。
4. 对图片或扫描件执行 OCR。
5. 根据知识库分段策略生成切片。
6. 调用 embedding 模型生成向量。
7. 将向量和可过滤 payload 写入 Qdrant。
8. 将文档、切片、处理状态和错误信息写入关系型数据库。
9. 更新文档状态为就绪或失败。

### 8.2 OCR 支持

系统需要支持图片 OCR 和扫描件 OCR。

OCR 场景包括：

- 图片文件直接上传。
- 扫描版 PDF。
- PDF、PPTX、DOCX 中嵌入的图片。
- 表格或截图中的文本。

要求：

- OCR 应作为可插拔解析能力，而不是写死在单一解析流程中。
- OCR 结果需要进入统一的切片和向量化流程。
- OCR 失败时应记录错误原因。
- 对扫描件或图片型 PDF，应能识别需要 OCR，而不是只返回空文本。

### 8.3 解析后端

系统应支持多种文档解析后端，并允许运行时切换。

解析后端可按类型区分：

- 文本解析器：处理 TXT、MD。
- Office 解析器：处理 DOCX、PPTX、XLSX。
- PDF 解析器：处理可提取文本 PDF。
- OCR 解析器：处理图片和扫描件。
- 复合解析器：根据文档内容自动选择文本提取或 OCR。

### 8.4 并发控制

文档处理管线应支持并发控制。

配置项包括：

- 最大并发解析任务数。
- 最大并发 OCR 任务数。
- 最大并发 embedding 请求数。
- 失败重试次数。
- 单任务超时时间。

并发控制的目的：

- 避免模型 API 过载。
- 避免 OCR 或解析任务耗尽服务器资源。
- 避免大量上传时影响在线问答。

## 9. Qdrant 向量存储与检索

向量数据库明确使用 Qdrant。

### 9.1 存储分工

建议采用关系型数据库 + Qdrant 的分工：

| 存储 | 职责 |
| --- | --- |
| 关系型数据库 | 用户、角色、知识库、文档、素材、切片元数据、会话、消息、配置、审计、统计 |
| Qdrant | 切片向量、向量检索索引、检索过滤 payload |
| 文件存储 | 原始文档、可能的解析中间产物 |

### 9.2 Qdrant Payload

Qdrant payload 应保存检索过滤和溯源所需字段。

建议包括：

- `knowledgeBaseId`
- `knowledgeBaseVisibility`
- `documentId`
- `documentName`
- `chunkId`
- `tags`
- `documentType`
- `assetType`
- `materialCategory`
- `pageNumber`
- `sectionPath`
- `chunkIndex`
- `createdAt`

### 9.3 权限过滤

检索权限必须在后端执行。

普通用户检索时：

- 仅允许检索 `public` 知识库。
- 请求 Qdrant 时必须带上可访问知识库 ID 或可见性过滤条件。
- 不能通过前端传参绕过权限限制。

管理员和超级管理员检索时：

- 可检索全部知识库。
- 管理员检索测试页面可以指定一个或多个知识库范围。

### 9.4 检索模式

系统应支持：

- 语义向量检索。
- 向量检索 + 重排序。

过滤能力：

- 知识库范围过滤。
- 文档标签过滤。
- 文档类型过滤。
- 相似度阈值过滤。
- 重排序分数阈值过滤。

结果字段：

- 文档名称。
- 知识库名称。
- 章节路径。
- 页码或位置。
- 匹配内容摘要。
- 相似度分数。
- 重排序分数。
- 标签。

## 10. 前台知识检索

系统应提供面向普通用户的知识检索页面。

功能包括：

- 输入自然语言查询。
- 选择一个或多个有权限的知识库。
- 按标签过滤。
- 选择检索模式。
- 设置或使用默认 Top K。
- 查看检索结果列表。

普通用户只能选择和检索 `public` 知识库。管理员可以选择全部知识库。

检索结果应展示：

- 文档名称。
- 知识库名称。
- 匹配内容摘要。
- 相关度分数。
- 标签。
- 来源位置。

如果用户没有任何可访问知识库，应给出明确空状态，而不是返回全局空结果或系统错误。

## 11. 智能问答

### 11.1 对话能力

智能问答模块应支持用户以自然语言进行交互。

功能包括：

- 新建对话。
- 发送问题。
- 接收流式回答。
- 接收非流式回答。
- 多轮上下文管理。
- 切换历史对话。
- 删除历史对话。
- 查看回答引用。
- 页面刷新后恢复会话。

### 11.2 SSE 流式输出

对话应支持 SSE 流式输出。

流式事件建议包括：

- 会话创建或确认事件。
- 意图识别结果。
- 检索开始。
- 检索结果摘要。
- 重排序完成。
- 生成开始。
- token 增量输出。
- 引用信息。
- 生成完成。
- 错误事件。

SSE 需要考虑：

- 客户端断开连接。
- 服务端生成超时。
- 模型 API 错误。
- 用户重复提交。
- 取消生成。

### 11.3 非流式输出

`docs/system.md` 中明确智能对话同时支持流式输出和非流式输出。非流式输出适合后台任务、自动评测、低复杂度调用或客户端不适合维护 SSE 连接的场景。

非流式接口要求：

- 请求和响应仍需完整记录到服务端会话历史。
- 返回结果中应包含回答正文、引用列表、检索摘要、使用配置和错误信息。
- 与 SSE 使用同一套意图识别、权限校验、检索、重排序、生成和引用溯源逻辑。
- 评测任务优先使用非流式接口，便于批量执行和结果对比。

### 11.4 多轮上下文

系统应维护多轮对话上下文。

要求：

- 每条用户消息和助手消息均保存到服务端数据库。
- 生成回答时可以引用当前会话的历史消息。
- 上下文长度需要受模型限制控制。
- 可以通过摘要、截断或最近消息窗口控制上下文规模。
- RAG 检索应主要基于当前问题，也可结合必要的会话上下文改写查询。

### 11.5 意图识别与路由

系统应自动识别用户意图，并路由到对应处理模块。

当前至少支持：

- 知识问答。
- 一般对话。

可预留：

- 前台知识检索。
- 报告生成。
- 数据分析。
- 无法判断，需要澄清。

路由原则：

- 与知识库内容相关的问题走 RAG。
- 普通闲聊或不需要知识库的问题可走一般对话。
- 用户无权限访问相关知识库时，应返回权限相关提示。
- 检索结果不足时，应允许模型基于事实不足进行拒答或提示补充问题，避免编造答案。
- 报告生成和数据分析属于可预留意图，当前不展开实现，但意图枚举和权限校验应允许后续接入。

### 11.6 意图访问控制

`docs/system.md` 要求不同角色拥有不同功能访问权限，因此意图路由不能只判断用户输入内容，还需要判断当前用户是否有权使用目标能力。

建议规则：

- 普通用户可使用一般对话、公开知识库问答和公开知识库检索。
- 管理员可使用所有知识库问答、检索测试、素材检索和后续报告生成相关意图。
- 超级管理员可使用全部能力，并可管理模型、配置和权限。
- 当用户命中无权限意图时，系统应返回权限提示，而不是静默降级到一般对话。

## 12. RAG 知识增强问答

### 12.1 RAG 主流程

RAG 问答流程：

1. 接收用户问题。
2. 校验用户身份和会话权限。
3. 识别用户意图。
4. 根据用户权限和问答配置确定检索知识库范围。
5. 对问题进行查询改写或向量化。
6. 在 Qdrant 中执行向量检索。
7. 按相似度阈值过滤。
8. 如启用重排序，则调用 rerank 模型。
9. 按重排序阈值过滤。
10. 组装上下文片段和引用信息。
11. 调用 LLM 生成回答。
12. 通过 SSE 返回生成过程、回答正文和引用。
13. 保存消息、检索结果、引用和配置快照。

### 12.2 准确率要求

需求说明书明确指出最终项目的问答准确率将作为考核指标。未来会提供标准问题集、标准答案和评分规则。

因此系统需要预留评测和复现能力：

- 保存每次问答使用的问题。
- 保存最终回答。
- 保存检索到的切片列表。
- 保存引用来源。
- 保存使用的知识库范围。
- 保存使用的模型配置快照。
- 保存 Top K、相似度阈值、重排序阈值等检索参数。
- 保存模型供应商和模型名称。
- 支持后续用标准问题集批量运行问答。
- 支持对比不同模型、不同切片策略、不同检索参数的效果。

### 12.3 无答案处理

当检索结果为空或相关性低时，系统应避免编造。

建议策略：

- 明确告知未在可访问知识库中找到足够依据。
- 展示尝试检索的知识库范围。
- 提示用户换一种问法或扩大检索范围。
- 管理员可在检索测试页面分析召回不足原因。

## 13. 引用溯源与原文下载

### 13.1 引用展示

系统生成的回答应支持引用标注。

每个引用应能展示：

- 原文片段。
- 来源文档名称。
- 所属知识库。
- 章节路径。
- 页码或位置。
- 相似度分数。
- 重排序分数。
- 切片类型。

连续多个引用来自同一文档时，应支持合并展示，减少重复引用噪音。

### 13.2 引用交互

建议支持：

- 引用编号点击查看详情。
- 引用详情弹窗。
- 悬浮预览。
- 点击固定。
- 原文片段预览。
- 长文本滚动预览。
- 来源文档下载入口。

引用交互需要兼顾快速查看和深度追溯：悬浮预览用于快速确认来源，点击固定用于阅读长片段或执行下载操作。

### 13.3 原文下载权限

原文下载需要权限校验和审计日志。

要求：

- 下载接口必须校验当前用户是否有权限访问该文档。
- 普通用户只能下载 `public` 知识库下的文档。
- 管理员和超级管理员可下载全部知识库文档。
- 不允许直接暴露服务器文件路径。
- 下载失败时应返回明确原因。

### 13.4 审计日志

文档下载需要记录审计日志。

建议字段：

- 用户 ID。
- 用户角色。
- 文档 ID。
- 知识库 ID。
- 操作类型，例如 `download_document`。
- 操作时间。
- IP。
- User-Agent。
- 操作结果。
- 失败原因。

配置变更、删除文档、批量删除知识库、重试处理任务等操作也建议纳入审计。

## 14. 思考过程展示

系统在生成回答时应展示处理链路，让用户获得及时反馈。

建议展示的是可审计的流程状态，而不是模型内部不可控的隐式推理。

可展示步骤：

- 识别问题意图。
- 确定检索范围。
- 检索相关文档。
- 重排序候选片段。
- 生成回答。
- 整理引用来源。

交互要求：

- 思考步骤支持折叠和展开。
- 流式输出过程中实时更新。
- 流式完成后默认自动折叠。
- 错误时展示失败步骤和可理解原因。

## 15. 管理后台

### 15.1 知识问答配置

管理员应可运行时配置问答模块参数。

配置项包括：

- 默认术语知识库。
- 默认技术监督知识库。
- 默认 Top K。
- 相似度阈值。
- 重排序阈值。
- 是否启用重排序。
- 问答提示词模板。
- 最大上下文片段数。
- 最大会话上下文长度。

### 15.2 检索体验测试

系统应提供检索体验测试页面，帮助管理员验证知识库召回效果。

功能包括：

- 输入测试问题。
- 选择一个或多个知识库。
- 设置 Top K。
- 设置相似度阈值。
- 开启或关闭重排序。
- 设置重排序阈值。
- 设置标签过滤。
- 查看召回片段。
- 查看相似度分数和重排序分数。
- 查看文档名称、章节路径、内容预览。

该页面对后续问答准确率调优非常关键。

### 15.3 模型配置

系统需要支持多供应商模型配置。

模型类型包括：

- LLM：用于问答生成和一般对话。
- Embedding：用于切片向量化和查询向量化。
- Rerank：用于检索结果重排序。
- OCR：用于图片和扫描件文字识别。

通用配置字段：

- 供应商类型。
- API Base URL。
- API Key。
- 模型名称。
- 超时时间。
- 是否启用。
- 额外参数 JSON。

供应商类型建议包括：

- `siliconflow`
- `openai-compatible`
- `local`
- `custom`

### 15.4 解析器与素材配置

管理后台还应覆盖文档解析器和素材管理相关配置。

解析器配置包括：

- 最大并发解析任务数。
- OCR 后端选择。
- 默认解析后端。
- 失败重试次数。
- 单任务超时时间。

素材管理配置包括：

- 素材分类字典。
- 常用标签键。
- 标签值约束。
- 报告类型与素材分类的推荐映射。
- 模板与素材关联规则。

### 15.5 配置运行时生效

模型配置、检索配置、解析器配置等应支持运行时生效，无需重启服务。

注意事项：

- 配置变更应即时影响新任务或新请求。
- 已在执行中的文档处理任务可以继续使用启动时配置，也可以按任务设计选择读取最新配置，但规则必须明确。
- 问答记录应保存当次使用的配置快照，便于后续复现和评测。

### 15.6 配置历史与回滚说明

配置历史版本是指每次管理员修改配置时，系统保留一份旧配置和新配置的记录，而不是简单覆盖。

价值：

- 可追踪谁在什么时候修改了配置。
- 可分析某次问答准确率下降是否由配置变更导致。
- 可复现历史问答使用的模型和参数。
- 可在配置效果变差时恢复到历史版本。

完整回滚 UI 可以后续实现，但建议当前数据设计至少预留：

- 配置版本号。
- 创建人。
- 创建时间。
- 生效状态。
- 变更原因。
- 配置快照。

## 16. 报告生成衔接

报告生成不是本文档的展开重点，但 `docs/system.md` 和 `docs/report_generation_system.md` 都表明报告生成会复用知识管理、素材管理、检索和模型能力。因此需要在当前设计中明确衔接边界。

### 16.1 报告生成依赖的上游能力

报告生成依赖：

- 用户认证和权限。
- 大语言模型配置。
- 知识库和素材文档管理。
- 文档解析、OCR、切片和向量化。
- Qdrant 检索和标签过滤。
- 模板与素材映射关系。
- 文件下载和审计能力。

### 16.2 报告生成需要复用的数据

报告生成模块后续可复用：

- 知识库中的规程规范、术语条目、技术报告论文。
- 素材管理中的厂级专业报告、技术文档、检查报告。
- 文档标签和素材键值标签。
- 章节路径、页码、切片类型和原文片段。
- LLM、Embedding、Rerank、OCR 的多供应商配置。

### 16.3 报告生成能力边界

后续报告生成模块将负责：

- 迎峰度夏检查报告和煤库存审计报告两类固定报告。
- 报告大纲生成。
- 大纲在线编辑、章节删除、顺序调整和自动编号。
- 逐章节流式内容生成。
- 正文、表格、图片内容处理。
- DOCX 导出和统一样式体系。
- 报告记录管理。
- 报告模板管理和可视化编辑。

当前知识管理与智能问答设计只需要保证：文档、素材、检索、模型配置和引用溯源能力足够稳定，后续报告生成可以复用这些能力，而不是另建一套平行系统。

### 16.4 模板与素材映射

报告模板管理需要建立模板与素材的映射关系。

建议预留：

- 模板 ID。
- 报告类型。
- 章节路径。
- 所需素材分类。
- 所需标签条件。
- 检索策略。
- 默认 Top K。
- 是否启用重排序。

这样后续生成某个章节时，可以根据模板章节自动检索对应素材和知识库内容。

## 19. 关键业务流程

### 19.1 文档入库流程

1. 管理员创建或选择知识库。
2. 管理员上传文档并添加标签。
3. 系统保存原始文件。
4. 系统创建文档记录和处理任务。
5. 解析器提取文本或结构化内容。
6. OCR 处理图片或扫描件。
7. 系统按知识库分段策略生成切片。
8. Embedding 模型生成向量。
9. 系统写入 Qdrant。
10. 系统更新文档状态为就绪。
11. 普通用户可在权限范围内检索和问答。

### 19.2 素材入库流程

1. 管理员进入素材管理页面。
2. 上传专业素材文档。
3. 选择素材分类并添加键值对标签。
4. 系统保存原始素材文件。
5. 系统复用文档处理管线解析、OCR、切片和向量化。
6. 系统将素材切片写入 Qdrant，并在 payload 中标记素材类型和标签。
7. 系统保存素材元数据。
8. 后续检索、问答或报告生成可在权限范围内召回素材。

### 19.3 智能问答流程

1. 用户进入问答页面。
2. 系统从服务端加载历史会话。
3. 用户发送问题。
4. 系统识别意图。
5. 系统按用户权限确定知识库范围。
6. 系统执行 Qdrant 检索和可选重排序。
7. 系统组装上下文和引用。
8. LLM 生成回答。
9. SSE 实时返回处理状态、正文和引用。
10. 系统保存会话消息、检索记录和引用。
11. 用户查看引用或下载原文。
12. 原文下载接口执行权限校验并写入审计日志。

### 19.4 报告生成衔接流程

1. 管理员上传或配置报告模板。
2. 管理员建立模板章节与素材分类、标签条件的映射。
3. 后续报告生成模块根据报告类型和章节需求发起素材检索。
4. 检索服务在 Qdrant 中召回相关知识库文档和素材切片。
5. LLM 基于模板、大纲、素材片段和知识库片段生成章节内容。
6. 报告生成模块保存生成内容、引用来源和导出记录。

### 19.5 配置变更流程

1. 管理员进入配置页面。
2. 修改模型、检索、解析器或问答参数。
3. 系统校验配置格式和连通性。
4. 系统保存新配置并记录操作人。
5. 新配置运行时生效。
6. 后续请求或任务使用新配置。
7. 问答和处理记录保存使用的配置快照。

## 20. 非功能需求

### 20.1 安全性

- 所有业务 API 需要认证。
- 后端必须执行权限校验。
- API Key 等敏感配置需要加密或安全存储。
- 原文下载不能暴露真实文件路径。
- 管理操作需要审计。
- 素材下载、模板修改和报告文件下载也应纳入权限校验和审计。

### 20.2 可用性

- 文档处理失败应允许重试。
- 模型供应商异常时应返回明确错误。
- SSE 断开时前端应能展示当前状态。
- 检索无结果时应给出明确提示。

### 20.3 可扩展性

- 模型供应商可扩展。
- 文档解析器可扩展。
- OCR 能力可扩展。
- 知识库权限模型可扩展。
- 报告生成模块可在后续复用知识检索、素材检索、引用溯源和模型配置能力。

### 20.4 可观测性

- 文档处理任务需要状态和错误信息。
- 问答需要保存检索轨迹。
- 模型调用需要记录耗时和失败原因。
- 配置变更需要记录历史。
- 素材处理和报告生成衔接需要记录检索轨迹。

## 21. 待确认问题

- 普通用户是否允许创建个人私有会话以外的个人知识库。
- `private` 知识库是否只对管理员可见，还是未来需要指定用户或部门可见。
- 原始文件存储使用本地磁盘、对象存储还是数据库外部文件服务。
- OCR 使用本地服务还是外部模型服务。
- 是否需要对文档下载做水印、限速或下载次数限制。
- 会话删除是软删除还是硬删除。
- 知识库和文档删除后，历史问答引用如何展示。
- 配置变更是否要求审批流程。
- 未来评测标准中是否包含引用正确性、拒答正确性和答案完整性。
- 素材是否与知识库共享可见性规则，还是需要单独的素材权限。
- 报告模板与素材映射由管理员手工配置，还是允许系统自动推荐。
- 报告生成是否要求对引用素材做正式出处清单。

## 22. 主要风险

- OCR 准确率会直接影响扫描件和图片文档的问答质量。
- 切片策略不合理会导致召回不足或上下文噪音过大。
- 多供应商模型接口差异可能导致配置和错误处理复杂化。
- 只在前端控制知识库可见性会造成数据越权风险，必须后端强校验。
- 不保存检索轨迹和配置快照会导致未来准确率考核难以复现问题。
- Qdrant payload 设计不足会影响权限过滤、标签过滤和引用溯源。
- 大文档批量上传和 OCR 任务可能造成处理队列积压，需要并发控制和监控。
- 素材管理如果与知识库文档处理割裂，会造成重复解析、重复向量化和后续报告生成口径不一致。
- 报告生成如果不复用现有检索和引用溯源能力，后续很难解释报告内容来源。

## 23. 结论

知识管理和智能问答应作为一个整体系统设计。知识管理负责把原始文档和专业素材转化为可检索、可追溯、可权限控制的知识资产；智能问答负责在权限范围内调用检索和模型能力，生成带引用、可复现、可评测的回答。报告生成后续应复用同一套知识资产、素材资产、检索服务和模型配置。

完整主链路是：

```text
用户认证
  -> 知识库管理
  -> 文档上传
  -> 素材上传
  -> 文档解析 / OCR
  -> 语义切片
  -> Embedding
  -> Qdrant 入库
  -> 权限过滤检索
  -> RAG 问答
  -> SSE 流式输出
  -> 引用溯源
  -> 原文下载审计
  -> 问答记录与评测复现
  -> 报告生成复用
```

后续架构设计和开发拆分应围绕这条主链路展开，优先保证数据入库质量、检索权限正确性、问答可复现性、引用可信度，以及报告生成对知识和素材的复用一致性。
